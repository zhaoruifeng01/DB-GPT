# DB-GPT 项目目录结构分析

## 项目概述

DB-GPT 是一个开源的 **Agentic AI 数据分析智能助手**，采用 Python monorepo + Next.js 前端架构。  
版本：0.8.1 | Python >= 3.10 | 包管理：uv workspace

---

## 核心目录

### `packages/` — Python 多包工作区（后端核心）

| 子包 | 用途 |
|------|------|
| `dbgpt-core` | 核心框架：Agent 系统、AWEL 工作流编排、RAG、多模型管理、数据存储抽象 |
| `dbgpt-app` | 应用入口：CLI 命令、Web 服务启动、配置管理 |
| `dbgpt-serve` | 模型服务层：本地/远程 LLM 部署与推理管理 |
| `dbgpt-ext` | 扩展模块：第三方集成、数据源连接器、工具插件 |
| `dbgpt-client` | Python SDK 客户端，用于与 DB-GPT API 通信 |
| `dbgpt-sandbox` | 沙箱环境：安全隔离地执行用户代码 |
| `dbgpt-accelerator` | 推理加速：含 dbgpt-acc-auto（自动加速）和 dbgpt-acc-flash-attn（Flash Attention） |

### `web/` — 前端应用（Next.js + TypeScript）

| 子目录 | 用途 |
|--------|------|
| `pages/` | 页面路由（传统 Pages Router） |
| `app/` | Next.js 13+ App Router |
| `components/` | 通用 React 组件库（14 个模块） |
| `new-components/` | 新版组件（chat、connector 等 9 个模块） |
| `hooks/` | 自定义 React Hooks |
| `client/api/` | API 客户端封装（11 个模块） |
| `types/` | TypeScript 类型定义 |
| `utils/` | 工具函数 |
| `locales/` | 国际化（中/英文） |
| `styles/` | 样式（Tailwind CSS） |
| `public/` | 静态资源 |

技术栈：Next.js、Ant Design、TypeScript、Tailwind CSS、@antv/g2

## 配置与部署

### `configs/` — 运行时配置模板

TOML 格式的预配置文件，按模型部署方式分类：

- **本地模型**：`dbgpt-local-glm.toml`、`dbgpt-local-qwen.toml`、`dbgpt-local-vllm.toml`、`dbgpt-local-mlx.toml`
- **代理/云模型**：`dbgpt-proxy-openai.toml`、`dbgpt-proxy-deepseek.toml`、`dbgpt-proxy-ollama.toml`、`dbgpt-proxy-tongyi.toml` 等
- **高级功能**：`dbgpt-graphrag.toml`（GraphRAG）、`dbgpt-bm25-rag.toml`（BM25 检索）
- **应用配置**：`dbgpt-app-config.example.toml`

### `docker/` — 容器化部署

| 子目录 | 用途 |
|--------|------|
| `base/` | 基础镜像 Dockerfile |
| `allinone/` | 一体化镜像（打包所有服务） |
| `compose_examples/` | Docker Compose 示例（单机/集群/高可用） |
| `datasources/` | 数据源配置（如 Hive） |
| `examples/` | SQL 初始化脚本、Dashboard 配置等 |

### `docker-compose.yml` — 默认编排

MySQL + WebServer 的最小化部署方案，使用 SiliconFlow 作为 LLM 代理。

## 架构层次总结

```
┌─────────────────────────────────────────────┐
│              web/ (Next.js 前端)              │
├─────────────────────────────────────────────┤
│            dbgpt-app (应用入口/API)           │
├─────────────────────────────────────────────┤
│   dbgpt-core (Agent/AWEL/RAG/多模型管理)      │
├──────────────┬──────────────┬───────────────┤
│  dbgpt-serve │  dbgpt-ext   │ dbgpt-sandbox │
│  (模型服务)   │  (扩展模块)   │  (沙箱执行)   │
├──────────────┴──────────────┴───────────────┤
│         dbgpt-accelerator (推理加速)          │
└─────────────────────────────────────────────┘
```

DB-GPT 后端架构分析
整体架构
DB-GPT 采用 monorepo 多包架构，基于 Python 3.10+，使用 uv 工作区管理 7 个核心子包：
包名	职责
dbgpt-core	核心框架：存储接口、Embedding工厂、文本分割、RAG基础类、AWEL编排
dbgpt-ext	可选扩展：向量数据库实现、文件存储（S3/OSS）、数据源适配、Embedding提供商
dbgpt-serve	业务服务层：RAG服务、模型管理、数据源连接、知识库管理
dbgpt-app	应用层：上层API、Agent场景、对话管理
dbgpt-client	客户端SDK
dbgpt-sandbox	代码沙箱执行环境
dbgpt-accelerator	模型推理加速
1. 数据库层
ORM框架：SQLAlchemy 2.0+，通过 DatabaseManager 统一管理
支持的数据库：MySQL（默认）、SQLite、PostgreSQL、DuckDB
连接池：基于 SQLAlchemy QueuePool，自动会话管理和事务提交
Schema定义：dbgpt.sql 定义了核心表结构（InnoDB + utf8mb4）：
knowledge_space - 知识空间
knowledge_document - 知识文档
document_chunk - 文档分块
connect_config - 数据源连接配置
chat_history - 对话历史
通用存储接口：SQLAlchemyStorage 实现了 StorageInterface 抽象接口，支持 CRUD + 分页
2. Embeddings
工厂模式：EmbeddingFactory 抽象类 + DefaultEmbeddingFactory 实现
支持的 Embedding 提供商（dbgpt-ext）：
OpenAI API（text-embedding-3-small 等）
HuggingFace / Sentence-Transformers（本地模型）
Ollama（本地部署）
通义千问（DashScope）
百度千帆
SiliconFlow
Jina AI
AIMLAPI
Rerank 支持：RerankEmbeddingFactory 用于检索后的重排序
3. 向量计算与向量数据库
有完整的向量计算支持，通过 VectorStoreBase 抽象接口统一管理，支持相似性搜索（similarity_search）。支持的向量数据库（均在 dbgpt_ext/storage/vector_store/ 目录）：
向量数据库	实现类	特点
ChromaDB	ChromaStore	默认选项，本地持久化
Milvus	MilvusStore	分布式，支持大规模数据
Weaviate	WeaviateStore	云原生
Qdrant	QdrantStore	高性能
PGVector	PGVectorStore	PostgreSQL扩展
Elasticsearch	ElasticStore	全文检索+向量混合
OceanBase	OceanBaseStore	分布式关系+向量
Valkey	ValkeyStore	Redis兼容
核心流程：EmbeddingAssembler → 调用 Embedding 模型生成向量 → 存入向量数据库 → EmbeddingRetriever 执行相似性搜索召回。
4. 存储层
文件/对象存储（dbgpt-ext）：
S3Storage - AWS S3 兼容存储（支持 MinIO 等）
OSS Storage - 阿里云 OSS 对象存储
统一存储接口：StorageInterface（抽象基类），通过 StorageItemAdapter 模式在业务对象和存储格式间转换。知识图谱存储：
TuGraph（默认图数据库）
Neo4j
支持 CommunitySummaryKnowledgeGraph（社区摘要）
5. RAG 架构
完整的 RAG 流水线，由 StorageManager 单例统一管理：
文档加载 → 文本分割 → Embedding生成 → 向量存储 → 相似性检索 → LLM生成
关键组件：
Knowledge：文档加载抽象（支持 PDF、Word、PPT、Excel、Markdown 等）
TextSplitter：文本分割器（chunk_size + chunk_overlap）
ChunkManager：分块管理器
EmbeddingAssembler：Embedding组装器，协调分块→向量化→持久化
EmbeddingRetriever：向量检索器
全文检索：Elasticsearch BM25（可与向量检索混合）
6. 架构设计模式总结
工厂模式：EmbeddingFactory、StorageManager
策略模式：TextSplitter、RetrieverStrategy 支持多种算法切换
适配器模式：StorageItemAdapter 在用户对象和存储格式之间转换
单例模式：StorageManager 全局唯一
插件系统：通过 pyproject.toml 的 optional-dependencies 实现可选集成（安装什么用什么）
