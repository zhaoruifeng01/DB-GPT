## 架构总览

### 包依赖（声明 vs 运行时）

**声明依赖（pyproject.toml，单向无环 DAG）**：
```
dbgpt-app ──┬─► dbgpt-serve ──► dbgpt-ext ──► dbgpt-core
            ├─► dbgpt-client ──► dbgpt-ext
            ├─► dbgpt-sandbox
            └─► dbgpt-acc-auto
```

**运行时导入（存在多层循环，靠 lazy import 掩盖）**：
- `dbgpt-core` 内 20+ 处函数级反向 import `dbgpt_serve`/`dbgpt_ext`/`dbgpt_app`
- `dbgpt-serve` 有 30+ 处反向 import `dbgpt_app`，**部分模块级非 lazy**
- `dbgpt-ext` 反向引用 `dbgpt_app`、`dbgpt_serve`

### 核心架构模式：混合架构（分层 + 组件化/DI + DAG 编排 + 工厂）

- **分层**：HTTP（api_v1/v2）-> Serve/Service -> Scene/AWEL -> LLM
- **组件化/DI**：`SystemApp` 作 IoC 容器，`BaseComponent` 注册/获取，Serve 模块可插拔（进程内组件，非独立服务）
- **DAG 编排（AWEL）**：`DAG`/`DAGNode`/`BaseOperator`/`WorkflowRunner`/`TaskContext` 抽象，`>>` 运算符 DSL 建立上下游，`DefaultWorkflowRunner` DFS 递归从叶节点回溯执行
- **工厂模式**：`ChatFactory` 按 `ChatScene` 枚举创建 chat 实现

### AWEL 关键抽象

- `DAG`（`dag/base.py:788`）：DAG 容器，管理节点映射、根/叶/触发节点
- `DAGNode`（`dag/base.py:297`）：通过 `>>`/`<<` 声明依赖
- `BaseOperator`（`operators/base.py:161`）：可执行 Operator，提供 `call()`/`call_stream()`
- `DefaultWorkflowRunner`（`runner/local_runner.py:24`）：默认执行器，`execute_workflow` 从叶节点 DFS 回溯
- Agent 被封装为 AWEL Operator：`WrappedAgentOperator`（`agent/core/plan/awel/agent_operator.py:56`）继承 `MapOperator`，在 `map()` 调用 `agent.generate_reply()`

### 典型请求流转（chat/completions）

入口 `api_v1.py:526` `chat_completions()` 按 `chat_mode` 分流：

- **ChatAgent**：`api_v1.py:555` -> `multi_agents.app_agent_chat()`（`agent/agents/controller.py:475`）-> Agent 执行 -> StreamingResponse
- **ChatFlow (AWEL)**：`api_v1.py:574` -> `FlowService.chat_stream_flow_str()`（`flow/service/service.py:436`）-> `safe_chat_stream_flow()`（`:525`）-> `_get_callable_task()` 从 `dag_manager.dag_map` 取叶节点 -> `safe_chat_stream_with_dag_task()`（`chat_util.py:80`）-> `task.call_stream()` -> `DefaultWorkflowRunner.execute_workflow()` -> StreamingResponse
- **ChatNormal/DB/Knowledge**：`api_v1.py:600` -> `get_chat_instance()`（`:469`）-> `ChatFactory.get_implementation()`（`chat_factory.py:9`）-> `BaseChat` 子类 -> LLMClient -> StreamingResponse

### Serve 加载

`register_serve_apps()`（`serve_initialization.py:81`）启动时逐个 `system_app.register(ServeClass)`；`Serve.init_app()`（如 `flow/serve.py:57`）通过 `app.include_router(router, prefix=...)` 挂载 FastAPI 路由。初始化链：`component_configs.py:14` -> `:58` `register_serve_apps()`。

## 优点（设计得当之处）

1. **AWEL DAG 编排抽象成熟**：`Operator`/`Runner`/`TaskContext` 分层清晰，`>>` 运算符 DSL 优雅；Agent 被统一封装为 Operator 编入 DAG，编排与执行解耦。（`dbgpt/core/awel/`）
2. **组件化/DI 骨架可插拔**：`SystemApp` + `BaseComponent` + `BaseServe` 让 serve 模块可插拔，支撑 controller/worker 分布式形态。
3. **LLM Provider 抽象分层**：`LLMModelAdapter`（ABC）+ `ProxyLLMClient` 统一 21 个云 provider 与本地推理（HF/vllm/llama.cpp/MLX），接入面广且边界清晰。（`dbgpt/model/adapter/base.py:121`、`dbgpt/model/proxy/base.py`）
4. **生态广**：多数据源（10+）、多向量存储（6）、多 DB 后端（11），集成层 `dbgpt-ext` 隔离扩展。
5. **依赖治理现代化**：uv workspace monorepo + `uv.lock` 锁定 + ruff，工程基础优于多数同类项目。

## 风险与问题

### P0（严重，阻碍演进/生产稳定性）

**P0-1 运行时循环依赖：core ↔ serve ↔ app ↔ ext，lazy import 掩盖**
- 证据：`dbgpt_serve/agent/agents/controller.py:39` `from dbgpt_app.dbgpt_server import system_app`（**模块级非 lazy**）；`dbgpt_serve/rag/operators/knowledge_space.py:25-27` 同样模块级导入 `dbgpt_app.knowledge.*`；`dbgpt_ext/rag/retriever/bm25.py:13` `from dbgpt_app.base import logger`；`dbgpt/core/_private/config.py:385` 等 20+ 处函数级反向 import。
- 影响：与 pyproject 声明直接矛盾；任何包无法独立安装/测试/部署；重构牵一发动全身；IDE 与静态分析无法准确解析依赖图。
- 建议：下沉共享 schema/接口到 `dbgpt-core`，serve 禁止模块级 import app；用抽象接口 + 注册表反转依赖；逐步将 lazy 反向 import 替换为显式扩展点。

**P0-2 同步 SQLAlchemy 阻塞 ASGI 事件循环**
- 证据：`db_manager.py:312` 使用 `create_engine`（同步），FastAPI 是 ASGI。
- 影响：高并发下 DB 访问阻塞事件循环，吞吐量受限；`scoped_session` 非真正异步。
- 建议：引入 async engine（SQLAlchemy 2.0 async + asyncpg/aiomysql），或至少用 `run_in_threadpool` 包装同步 DB 调用。

**P0-3 任务调度非持久化**
- 证据：`dbgpt-serve/scheduled_task/serve.py:31` 明确使用 `MemoryJobStore`。
- 影响：单点重启丢失所有调度任务；分布式场景无跨节点协调。
- 建议：切换 `SQLAlchemyJobStore` 或 `RedisJobStore`；分布式场景引入 leader 选举或分区。

### P1（重要技术债）

**P1-1 上帝包：dbgpt-core 承担 8+ 不相关领域**
- 证据：`src/dbgpt/` 下并列 `core/rag/model/agent/datasource/storage/vis/train/util/cli/_private/configs/experimental/component.py`。
- 影响：单包体积过大，认知负担高；版本耦合，任何子域变更触发整包发版；不利于独立复用。
- 建议：拆分为 `dbgpt-awel` / `dbgpt-agent` / `dbgpt-rag` / `dbgpt-model` / `dbgpt-datasource` / `dbgpt-storage` 独立包。

**P1-2 dbgpt-serve 过载（17 个并列子模块）**
- 证据：`dbgpt_serve/` 下 agent/connector/conversation/core/datasource/dbgpts/evaluate/feedback/file/flow/governance/libro/model/permission/prompt/rag/scheduled_task/utils。
- 影响："所有 serve 功能总和"而非单一职责；与 app 层 scene/agent 职责重叠。
- 建议：按业务域聚合，长期可独立部署（serve-flow / serve-agent / serve-rag）。

**P1-3 上帝函数：`chat_completions()` 混合 110 行逻辑**
- 证据：`api_v1.py:526`，路由函数内做模式分支、领域类型解析、流式包装、异常处理、用量记录。
- 影响：难以测试、难以扩展新模式、违反 SRP。
- 建议：抽 `ChatModeDispatcher`，每个 mode 独立 handler。

**P1-4 Metrics 完全缺失**
- 证据：无 Prometheus exporter、无 `/metrics` 端点。
- 影响：生产可观测性不完整（有 tracing + 日志但无指标）；无法量化 QPS/延迟/错误率。
- 建议：引入 `prometheus-fastapi-instrumentator`，暴露 `/metrics`。

**P1-5 mypy 声明但未强制**
- 证据：`mypy>=1.15` 在 dev-deps，但 `.pre-commit-config.yaml:32-40` 中 mypy hook 被注释；`pyproject.toml` 无 `[tool.mypy]`。
- 影响：类型安全形同虚设，重构无保障。
- 建议：启用 mypy（至少 core 子包），逐步收紧 strictness。

**P1-6 脆弱反射注册：`ChatFactory` 依赖 `__subclasses__()` + import 副作用**
- 证据：`chat_factory.py:48` 用 `BaseChat.__subclasses__()` 查找实现；依赖所有场景模块在调用前被 import。
- 影响：新增场景忘记在工厂添加 import 即静默失败；import 顺序敏感。
- 建议：改显式装饰器注册（`@register_chat(ChatScene.XXX)`）。

**P1-7 隐式耦合：DAGVar 全局上下文 + 字符串键字典传值**
- 证据：`dag/base.py:136` `DAGVar` 用 `threading.local`+`contextvars` 传 system_app/executor/dag；`dag/base.py:725` `DAGContext._share_data` 字典传值，键为字符串拼接（`_build_task_key`，`:527`）。
- 影响：节点间状态无类型安全，重构易引入 typo 静默失败。
- 建议：用 Pydantic models 或 TypedDict 替代字符串键字典。

**P1-8 `_private` 跨包公共化**
- 证据：`dbgpt._private` 被 20+ 处跨包导入（serve/flow/api/schemas.py:3、app/scene/base.py:4、core/model/cluster/base.py:3）。
- 影响：命名"私有"实为公共 API，语义误导，难以重构。
- 建议：改名为 `dbgpt.core.compat` 等公开命名。

**P1-9 Serve 生命周期隐式顺序依赖**
- 证据：`component_configs.py:52-58` 中 AWEL 初始化（`_initialize_awel`）在 `register_serve_apps` 之前，FlowServe 依赖 `dag_manager.dag_map`；顺序靠代码行序。
- 影响：初始化顺序错误将运行时才暴露问题；无显式依赖声明。
- 建议：引入 component 依赖图（声明 `depends_on`），拓扑排序初始化。

## 系统上下文

DB-GPT 是面向数据分析的 Agentic AI 平台：浏览器通过 Next.js 静态 UI 和 HTTP API 发起请求，Python FastAPI WebServer 将请求分发到聊天、知识库、工作流、Agent、数据源及治理模块；运行时通过 `SystemApp` 进程内组件容器装配能力，并持久化元数据到 SQLite/MySQL/OceanBase，RAG 可选接入 Chroma 与图数据库。现有 docker-compose 的默认形态是 MySQL + 单个 WebServer，集群和 HA 编排仅作为独立示例存在。

| 层次 | 主要技术 | 架构判断 |
| --- | --- | --- |
| 前端 | Next.js 13、React、TypeScript | 前后端可独立开发，生产可静态导出；构建期忽略 TypeScript 错误。 |
| 应用边界 | FastAPI / Uvicorn / `SystemApp` | 模块化单体，服务模块是进程内组件，不是默认独立微服务。 |
| 业务与 AI | AWEL、Agent、LLM provider、RAG | 插件化良好，扩展面广，启动装配和运行期依赖较复杂。 |
| 数据 | SQLAlchemy、SQLite/MySQL/OceanBase、向量/图存储 | 元数据与推理/检索可横向扩展，但默认单节点路径仍是本地状态优先。 |
| 部署与安全 | TOML、Docker Compose、JWT / API key | 配置模板以本地易用性为优先，不适合作为生产安全基线。 |

## 架构总览

识别出的模式是“分层 + 插件化组件容器 + DAG 工作流”的模块化单体，而非微服务系统。`dbgpt-app` 创建 FastAPI 和 `SystemApp`，挂载 v1/v2/知识库/Agent 等路由（`packages/dbgpt-app/src/dbgpt_app/dbgpt_server.py:52-97`）。启动先完成数据库、路由、迁移、权限服务等关键步骤（`:207-243`），再在后台线程注册重型组件、模型 worker 和服务（`:328-403`）。每个 `Serve` 模块在初始化时向同一 FastAPI 实例挂载路由，例如 Flow（`packages/dbgpt-serve/src/dbgpt_serve/flow/serve.py:59-65`）与 Governance（`packages/dbgpt-serve/src/dbgpt_serve/governance/serve.py:49-53`）。

关键数据流（文字时序）：

`浏览器 -> Next.js 静态资源 / API -> FastAPI Router -> Depends(get_user_from_headers) -> SystemApp 组件/Serve -> SQLAlchemy 元数据与数据源/RAG 存储 -> 模型或沙箱 -> 流式 HTTP 响应`。

实际授权分流不在统一 API 网关或全局 FastAPI middleware，而是分散在端点依赖和各 Serve 的可选 API key 检查中。`get_user_from_headers` 被聊天、知识库、Agent 和上传等多个端点直接依赖（如 `packages/dbgpt-app/src/dbgpt_app/openapi/api_v1/api_v1.py:183`、`packages/dbgpt-app/src/dbgpt_app/openapi/api_v1/python_upload_api.py:33`）。

## 优点

1. **模块化单体边界务实**：Python uv workspace 清楚分出 core、app、serve、ext、sandbox 和 client；默认部署没有过早拆成网络微服务，利于本地调试和快速迭代（`pyproject.toml:21-30`）。
2. **装配与启动路径显式**：关键初始化、延迟初始化和就绪端点都有明确代码边界（`packages/dbgpt-app/src/dbgpt_app/dbgpt_server.py:207-243`、`:328-403`、`:166-203`），便于将来迁移到受管容器平台。
3. **治理能力已作为独立业务模块进入进程内架构**：权限、治理模型和治理路由已由 `serve_initialization` 注册，而不是仅存在于 UI（`packages/dbgpt-app/src/dbgpt_app/initialization/serve_initialization.py:372-385`）。
4. **配置支持外部化模型密钥**：样例对 OpenAI 等模型密钥采用 `${env:...}`，避免把云模型 API key 直接写死（`configs/dbgpt-app-config.example.toml:37-48`）。

## 风险与问题

### P0

**P0-1 默认认证会授予匿名调用方管理员身份，且默认配置将服务绑定至全部网卡。**

- 问题描述：认证辅助函数在未显式设置 `DBGPT_AUTH_TYPE=db` 时默认进入 `mock` 模式；即使没有 `user_id`，也构造 `user_id="001"`、`role="admin"` 的用户。多份常用 TOML 又同时设置 `api_keys = []` 与 `host = "0.0.0.0"`。
- 证据：`packages/dbgpt-serve/src/dbgpt_serve/utils/auth.py:33-74`；`configs/dbgpt-app-config.example.toml:5-11`；63 个配置文件含有空 `api_keys`。
- 影响：一旦按示例或默认配置对内网/公网暴露，聊天、数据源、文件上传、Agent 等依赖此函数的 API 可被匿名调用并获得管理员语义，存在数据泄露、任意分析任务提交和配置篡改风险。
- 建议：生产模式下删除 mock fallback，改为 fail-closed 的全局认证 middleware；仅在显式 `DBGPT_AUTH_TYPE=mock` 且 `ENV=development` 时启用 mock。将绑定地址默认为 `127.0.0.1`，生产 profile 强制非空 API key 或 OIDC/JWT 配置，并在启动时校验失败即退出。

**P0-2 JWT 认证失败会降级到 mock 身份，且签发与校验未共享同一个密钥配置源。**

- 问题描述：登录签发 token 使用 Permission Serve 的 `jwt_secret_key` 配置；请求校验却只读取 `DBGPT_JWT_SECRET_KEY` 环境变量，并在任何异常后回退 mock 用户。
- 证据：`packages/dbgpt-serve/src/dbgpt_serve/permission/service/service.py:147-182`；`packages/dbgpt-serve/src/dbgpt_serve/permission/config.py:19-25`；`packages/dbgpt-serve/src/dbgpt_serve/utils/auth.py:35-74`。
- 影响：当配置文件为 JWT 设置密钥、但没有同步环境变量时，合法 token 可能验证失败；由于异常路径继续返回管理员 mock 用户，认证故障会转化为权限提升，而不是 401/403。
- 建议：将 token 验证收敛为单一 `AuthProvider`，从已经初始化的 `PermissionService` 获取配置；删除静态默认密钥和异常降级路径。新增“错误密钥、过期 token、无 token、服务未初始化”四类集成测试，并把未认证访问固定为 401。

**P0-3 服务 API key 为空时明确允许访问，形成第二条可绕过的授权链路。**

- 问题描述：Flow 与 Conversation Serve 在 API key 未设置时直接返回允许访问；并对 `/api/v1` 路径跳过该检查。
- 证据：`packages/dbgpt-serve/src/dbgpt_serve/flow/api/endpoints.py:85-106`；`packages/dbgpt-serve/src/dbgpt_serve/conversation/api/endpoints.py:70-90`；`configs/dbgpt-app-config.example.toml:5`。
- 影响：即使修复部分端点的用户依赖，仍可能因服务级可选认证和路径例外出现未经统一策略审查的访问面。授权模型不可审计，也无法验证治理规则是否全链路生效。
- 建议：定义唯一的认证/授权策略层：浏览器用户采用 OIDC/JWT，服务到服务采用 scoped API key；路由只声明所需 scope，禁止“空配置即放行”。在 CI 中扫描公开路由和豁免路径，要求有显式安全理由。

### P1

**P1-1 CORS 对任意源开放并允许携带凭据。**

- 问题描述：Uvicorn 外层 CORS 配置使用 `allow_origins=["*"]` 与 `allow_credentials=True`。
- 证据：`packages/dbgpt-app/src/dbgpt_app/dbgpt_server.py:449-456`。
- 影响：跨域边界难以审计，随着 cookie/session 或浏览器凭据策略演进，容易扩大 CSRF 和数据读取风险；也不符合生产环境最小暴露原则。
- 建议：配置化 allowlist，仅允许已部署的 Web UI 域名；区分开发、预发和生产 profile，生产启动时拒绝通配源与 credentials 的组合。

**P1-2 后台初始化失败仍将应用标记为 ready，健康检查无法表达降级或依赖故障。**

- 问题描述：模型、AWEL、Serve 等重型组件在后台线程初始化；捕获任何异常后依旧在 `finally` 中设置 `_app_ready`。
- 证据：`packages/dbgpt-app/src/dbgpt_app/dbgpt_server.py:328-403`。
- 影响：负载均衡器会将部分能力不可用的实例接入流量，用户请求随后才以组件缺失或运行时错误失败；故障恢复和滚动发布不可靠。
- 建议：拆分 liveness / readiness / dependency health；只有必需组件、认证服务、模型路由和数据源健康后才返回 ready。将初始化状态和错误原因结构化输出，并由编排平台按 readiness 执行流量切换。

**P1-3 前端构建忽略 TypeScript 类型错误，削弱前后端 API 演进的最后一道静态契约。**

- 问题描述：Next.js 配置明确开启 `ignoreBuildErrors`。
- 证据：`web/next.config.js:9-11`。
- 影响：后端 schema 或 API 变更可在没有前端编译失败的情况下发布，错误滞后到用户运行期；对多模块单体尤其容易引发兼容性回归。
- 建议：先在 CI 单独执行 `tsc --noEmit`，将现存错误建账后分批清零；最终移除 `ignoreBuildErrors`，并从 OpenAPI 或共享 schema 生成客户端类型。

### P2

**P2-1 配置模板含明文图数据库密码与通用默认加密密钥，易被直接复制到环境。**

- 问题描述：GraphRAG 示例包含 TuGraph 用户名/密码；多份 profile 使用 `encrypt_key = "your_secret_key"`。
- 证据：`configs/dbgpt-graphrag.toml:32-37`；`configs/dbgpt-app-config.example.toml:1-6`。
- 影响：模板被当作可运行部署配置时，弱密钥和可公开的密码会带来可预测凭据风险。
- 建议：所有 secret 改为 `${env:...}` 且启动时执行强度/非默认值校验；示例密码替换为不可用占位符，提供 `.env.example` 和密钥轮换说明。

**P2-2 单进程组件容器与“集群/HA 示例”之间缺少可验证的生产运行契约。**

- 问题描述：默认启动路径在同一进程中注册组件并初始化本地模型 worker；项目同时提供 cluster/HA compose 示例，但没有在默认部署文件中体现实例无状态化、会话粘性、任务协调和就绪探针约束。
- 证据：`packages/dbgpt-app/src/dbgpt_app/dbgpt_server.py:342-389`；`docker/compose_examples/cluster-docker-compose.yml`；`docker/compose_examples/ha-cluster-docker-compose.yml`。
- 影响：从开发单机迁移到多副本时，模型路由、后台任务、文件状态和会话一致性可能出现行为差异，运维团队缺少可验收的架构基线。
- 建议：先把 WebServer 定义为无状态 API 节点，将共享状态外置到数据库/对象存储/队列；为 worker、scheduler、Web API 明确部署拓扑与 SLA，再以 branch-by-abstraction 抽象本地/远程 worker 路由，逐步替换单进程耦合。

## 改进建议与现代化路径

### 短期（0-2 周）

1. 发布安全默认值：生产 profile 强制 `auth_type=db` 或 OIDC，禁止空 API key、mock admin、默认 JWT/encrypt key 和 `0.0.0.0` 裸暴露。
2. 用单一认证依赖替代分散的 `get_user_from_headers` 与各 Serve 的可选检查；加入安全回归测试矩阵，并将其设为发布阻断项。
3. 将 CORS 改为 profile 化 allowlist；对 API 增加审计日志，记录主体、数据源、Agent/skill、结果状态和请求链路 ID。
4. 修正 readiness 语义，初始化失败时保持 not-ready，并提供可观测的失败原因。

### 中期（1-2 个月）

1. 用 branch-by-abstraction 建立 `AuthProvider`、`SecretProvider` 和 `WorkerRouter` 接口：旧 JWT/mock 和本地 worker 作为适配器，逐个路由迁移后删除不安全实现。
2. 从 OpenAPI 或共享 schema 生成 TypeScript client，启用 `tsc --noEmit`，再删除 `ignoreBuildErrors`。
3. 把模型调用、文件/沙箱任务和调度任务的状态外置；为失败重试、幂等键和死信/人工干预定义统一约定。

### 长期（3-6 个月）

1. 采用 strangler 模式将高风险执行面拆出：先分离 sandbox/job runner，再分离模型 worker；Web API 保持 BFF/控制面职责，内部服务使用显式契约和 mTLS/服务身份。
2. 制定生产架构 fitness functions：未认证请求必须 401、默认密钥不得启动、ready 必须包含关键依赖、每个敏感 API 必须产生日志、每个配置 profile 必须通过安全基线测试。
3. 建立 ADR 和部署参考实现，明确单机、团队部署与 HA 的支持边界及升级路径。

## 风险与问题

### P0

本次样式换新未发现需要停止开发的 P0 问题。

### P1

**P1-1 设计令牌失控**
- 问题：全局 Ant 主题仅有 `colorPrimary` 和 `borderRadius`，Tailwind 又维护另一套颜色。
- 证据：`web/pages/_app.tsx:114-120`；`web/tailwind.config.js:16-34`。
- 影响：换肤、暗色、密度和品牌升级无法全局一次生效。
- 建议：建立 semantic token 单一真源，同时映射 Ant Token 与 Tailwind CSS variables。

**P1-2 直接覆盖 Ant 内部 DOM/CSS 选择器**
- 问题：全局 CSS 依赖 `.ant-*` 及生成的 `css-dev-only-do-not-override-*` 选择器。
- 证据：`web/styles/globals.css:31-53,79-85`；`web/new-components/layout/style.css:3-18`。
- 影响：Ant 升级或 DOM 调整容易导致样式静默失效；v6 已明确调整多个组件 DOM。
- 建议：逐页迁移到 global/component token、`classNames`/`styles` 组件 API，删除哈希类选择器。

**P1-3 多 UI 样式引擎造成心智和体积负担**
- 问题：Ant Design + MUI/Joy + Emotion + Tailwind + 全局 CSS 并存。
- 证据：`web/package.json:25-47,88-89`；MUI 仅分布在 `components/chart/*` 和 `components/chat/*` 5 个文件。
- 影响：视觉状态、暗色逻辑、弹层层级和包体积难以统一。
- 建议：先用 Ant 替换 5 个 MUI 使用点，移除 Joy/Material/Emotion；新组件限制为 Ant + Tailwind layout。

**P1-4 大量具体颜色和圆角直写到 JSX**
- 问题：约 1784 个 Tailwind 具体颜色类，151 个 `rounded-xl/2xl/3xl/full` 类使用点。
- 证据：代码扫描 `web/app`、`web/components`、`web/new-components`。
- 影响：改色并不等于换新，视觉容易继续不一致。

## 系统上下文

DB-GPT 当前采用模块化单体和进程内 Serve 组件模式。治理模块已经注册进主应用，并复用 `connect_config`、ConnectorManager 与 Permission Serve，但仍处于功能原型阶段。云枢具备更完整的语义元数据、权限聚合、审计和分钟指标设计，但其实现依赖独立 FastAPI 生命周期、MySQL 原生表、Redis、APScheduler、独立用户/数据源和 SQL Lab，不能直接嵌入。

### 技术栈概要

| 领域 | DB-GPT 当前技术 | 云枢可参考能力 | 改造选择 |
| --- | --- | --- | --- |
| Web/API | FastAPI + BaseServe + SystemApp | FastAPI Portal Router | 保留单一 DB-GPT 应用，不挂载云枢应用 |
| 元数据库 | SQLAlchemy Model + Alembic | aiomysql + 版本 SQL | SQLAlchemy Entity/DAO + Alembic |
| 数据源 | `connect_config` + ConnectorManager | `sys_data_source` + 自建异步池 | 只使用 DB-GPT 数据源体系 |
| 身份权限 | `sys_user/sys_role/sys_dept` + JWT | `api_users` + 角色/直接权限 + Redis 缓存 | DB-GPT 身份为唯一源，重建结构化 Grant |
| SQL 解析 | `sqlparse==0.4.4` + 前缀/正则 | `sqlparse==0.5.5` + Adapter 校验 | 升级 0.5.5，建立共享 SQL Guard |
| 可观测性 | 自有 Tracer + OTel Exporter | 审计、分钟聚合、psutil/Redis 状态 | OTel + Prometheus 指标 + 脱敏审计 |
| 前端 | React/Next.js；当前治理另有 Vue 微前端 | Vue 3 Portal | 短期可兼容，长期归入 DB-GPT 原生 React |

## 架构总览

推荐采用模块化单体内的领域分层，并使用 Branch by Abstraction 渐进替换现有 GovernanceService：

1. API 从 Permission Serve 获取统一 Principal（`permission/api/endpoints.py:56-208`）。
2. 元数据通过 Datasource Adapter 引用 `connect_config.id` 和 ConnectorManager（`datasource/manages/connect_config_db.py:28-55`）。
3. SQL 执行先经过 SQL Guard 提取资源，再由统一 Authorizer 聚合 user/role/dept 权限；现有首 Token + 正则逻辑位于 `governance/service.py:95-109`，应被适配器逐步替换。
4. 执行成功后进行列级过滤和脱敏，再异步投递审计事件和治理指标；现有同步审计位于 `governance/service.py:456-473`。
5. OTel Span 串联 auth、guard、authorize、connect、execute、mask、audit；分钟 Worker 只做聚合和留存。

目标模块边界为 `metadata`、`policy`、`sql_guard`、`audit`、`observability` 和薄 `api`。Serve 只负责装配，不再在启动阶段执行 `entity.__table__.create`（现位于 `governance/serve.py:58-90`）。

## 优点

1. **现有集成入口正确**：Governance 已按 BaseServe 嵌入主应用，无需新增第二个 Web 进程。
2. **数据源复用方向正确**：治理策略引用 `connect_config.id`（`governance/models.py:10-25`），可避免双数据源主数据。
3. **身份模型基础具备**：用户、角色、部门及其关联已存在（`permission/models/models.py:20-112`），适合扩展 Subject 聚合。
4. **连接器和 Tracing 底座可复用**：无需移植云枢数据库 Adapter，也无需新建追踪体系。
5. **SQL Lab 可完整切除**：目标能力能通过领域服务和执行拦截点独立存在，不要求 SQL 编辑器或 Lab Router。

## 风险与问题

### P0

**P0-1 身份管理端点未统一授权**

- 证据：`permission/api/endpoints.py:93-208` 的用户、角色、部门 CRUD 未声明认证/管理员依赖。
- 影响：攻击者可能修改身份和角色，使任何后续细粒度 RBAC 失效。
- 建议：除登录外统一 Principal；管理写操作要求 admin 或 `permission.manage`，将 401/403 测试设为发布阻断。

**P0-2 生产默认 JWT 密钥**

- 证据：`permission/config.py:19-25` 默认值为固定 HS256 密钥。
- 影响：沿用默认配置时身份可被伪造。
- 建议：生产检测到默认/空值时拒绝启动，密钥来自受控 Secret Provider。

**P0-3 SQL Guard 无法作为安全边界**

- 证据：`governance/service.py:95-117` 使用首 Token、正则提表和字符串拼接 LIMIT，Connector 最终可执行任意 SQL。
- 影响：复杂 CTE、子查询、方言和畸形输入可能误判，导致越权或写操作风险。
- 建议：升级 `sqlparse==0.5.5`，默认拒绝，建立方言语料并使用数据库只读凭据兜底；接入所有 SQL 执行入口。

**P0-4 审计敏感信息风险**

- 证据：云枢 `app/core/middleware.py:92-170` 会捕获请求体、响应体和 source_sql；该模式若直接移植会记录密钥和业务数据。
- 影响：审计系统本身成为敏感数据集中泄漏面。
- 建议：仅迁移指标定义，不复制中间件；审计采用字段白名单、递归脱敏和 SQL 指纹。

### P1

**P1-1 运行时 DDL 绕过迁移治理**：`governance/serve.py:58-90` 启动时自动建表。建议所有新旧治理表纳入 Alembic，启动只做版本检查。

**P1-2 GovernanceService 职责过载**：查询、授权、限流、脱敏、目录、API Key、审计集中在一个服务。建议使用接口和旧适配器渐进拆分，不做一次性重写。

**P1-3 请求路径频繁查库**：`governance/service.py:391-412` 每次查询读取 Grant，并通过审计表 COUNT 限流。建议权限版本缓存和令牌桶限流。

**P1-4 审计同步提交**：`governance/service.py:456-473` 在请求路径提交审计事务。建议有界队列、批量写入、背压和关键变更强审计。

**P1-5 SQL 执行入口可能遗漏**：仅保护 `/governance/query` 无法覆盖 Agent、Tool 和内部数据库访问。建议先建立入口清册，未接入的入口默认关闭。

**P1-6 数据模型引用松散**：当前 Grant 使用 `role_code + table_pattern`（`governance/models.py:28-47`）。建议改为 subject_type/id、结构化资源、action、effect 和有效期。

### P2

**P2-1 双前端技术栈**：治理 Vue 微前端与 DB-GPT React/Next.js 并存。短期可兼容，长期应使用 Strangler 模式迁入主前端。

**P2-2 画像与 AI 耦合风险**：云枢画像增强调用独立 AIService。基础画像必须不依赖 LLM，AI 增强作为默认关闭的可选 Adapter。
